# Animal Generation Canvas

Одностраничное приложение для визуального ведения родословной животных на интерактивном канвасе.

Документ ниже описывает проект как спецификацию для реализации с нуля. Если разработчик не видел исходный код, этого README должно хватить, чтобы восстановить текущее поведение, архитектурные ограничения и важные UX-детали.

## 1. Назначение

Приложение предназначено для ведения семейного дерева животных.

Ключевая доменная оговорка:

- один канвас = один вид животных
- на одном канвасе не смешиваются разные виды
- при этом сам проект не должен быть зашит под конкретный вид
- вид животного не хранится в каждом узле

Иными словами, приложение универсально по отношению к виду, но каждый конкретный набор данных описывает только одно семейство одного вида.

## 2. Технологический стек

- React 18
- TypeScript
- Vite
- Canvas 2D API для рендера дерева
- локальное хранение в `localStorage`

UI строится поверх React, но само дерево рисуется на `canvas`, а не как DOM/SVG-структура.

## 3. Главные пользовательские сценарии

Пользователь должен иметь возможность:

- просматривать дерево поколений на бесконечном поле
- перемещать поле мышью
- менять масштаб колесом мыши
- выбирать будущих родителей прямо с канваса
- создавать новое животное с уже предзаполненными родителями
- редактировать существующее животное
- удалять животное
- быстро вернуть вид канваса в стартовое состояние
- переключать язык интерфейса между `ru` и `en`
- открыть popup-подсказку по использованию приложения

## 4. Модель данных

### 4.1. Сущность узла

Базовая сущность:

```ts
type AnimalGender = "male" | "female";

type Animal = {
  id: string;
  name: string;
  gender: AnimalGender;
  fatherId: string | null;
  motherId: string | null;
  birthDate: string;
};
```

Требования к полям:

- `id` уникален в рамках канваса
- `name` обязателен
- `gender` обязателен и ограничен `male | female`
- `fatherId` может ссылаться только на существующий узел мужского пола
- `motherId` может ссылаться только на существующий узел женского пола
- `birthDate` хранится как строка формата `YYYY-MM-DD`

### 4.2. Ограничения данных

- животное не может быть собственным отцом
- животное не может быть собственной матерью
- отец и мать не могут быть одним и тем же животным
- если узел удаляется, ссылки на него у потомков очищаются
- если у животного есть потомки, пол в форме редактирования менять нельзя
- пустые или битые parent-ссылки должны нормализоваться в `null`
- дубликаты `id` при чтении данных должны отбрасываться

### 4.3. Формат хранения

Текущий формат snapshot:

```ts
type AnimalSnapshotV1 = {
  version: 1;
  animals: Animal[];
};
```

`localStorage`:

- ключ данных дерева: `animal-generation-canvas`
- ключ языка: `animal-generation-locale`

### 4.4. Генерация id

Новые узлы создаются как:

```ts
animal-${crypto.randomUUID()}
```

## 5. Архитектурные требования

### 5.1. Разделение слоёв

Проект должен быть разделён минимум на два логических слоя:

1. UI и взаимодействие с канвасом
2. слой данных и сериализации

Слой данных должен быть изолирован от React-компонента и отвечать за:

- типы `Animal` и `AnimalGender`
- создание id
- создание snapshot
- парсинг snapshot
- загрузку из `localStorage`
- сохранение в `localStorage`
- нормализацию данных

Причина:

- в будущем данные потребуется импортировать и экспортировать через файл, сеть или другой transport
- UI не должен знать детали сериализации

### 5.2. Точка расширения на будущее

Если позже появится import/export, он должен строиться поверх уже существующего data-layer, а не через парсинг JSON прямо в UI.

## 6. Layout дерева

### 6.1. Общая логика

Узлы группируются по поколениям.

Поколение вычисляется рекурсивно:

- если у узла нет родителей, поколение = 0
- иначе поколение = `max(поколение родителей) + 1`

Для layout используются:

- `NODE_WIDTH = 172`
- `NODE_HEIGHT = 90`
- `GENERATION_GAP = 260`
- `SIBLING_GAP = 120`
- `COMPONENT_GAP = 180`

### 6.2. Компоненты связности

Если в данных несколько несвязанных семей, они должны отображаться как отдельные компоненты, расположенные вертикально одна под другой с промежутком `COMPONENT_GAP`.

### 6.3. Порядок внутри поколения

Сортировка внутри layout:

1. по поколению
2. по `birthDate`
3. по исходному порядку массива

### 6.4. Базовая точка канваса

Центр дерева не должен быть ровно по центру экрана.

Текущая опорная точка:

- `x = width * 0.25`
- `y = height / 2`

Это значит:

- дерево стартует левее классического центра
- слева не должно оставаться пол-экрана пустоты
- одна и та же опорная точка обязана использоваться в рендере, hit-testing и zoom-логике

## 7. Визуальный вид узла

Каждый узел выглядит как белая карточка с мягкой тенью.

Внутри карточки:

- цветная точка пола слева
- имя
- подпись поколения
- дата рождения
- маленькая кнопка-шестерёнка в правом нижнем углу

Цвет пола:

- male: голубой акцент
- female: розовый акцент

### 7.1. Выбранные родители

Если узел выбран как родитель, карточка меняет белый фон на желтоватый.

### 7.2. Hover-состояния

Обязательны отдельные hover-состояния для двух разных зон:

- тела карточки
- шестерёнки

При hover:

- над карточкой и шестерёнкой курсор должен становиться `pointer`
- карточка должна отдельно подсвечиваться
- шестерёнка должна отдельно менять цвет/обводку/тень
- пользователь должен визуально понимать, что это две разные кликабельные области

### 7.3. Попадание по шестерёнке

Шестерёнка имеет собственную круглую hit-zone.

Текущий радиус: `12px`.

## 8. Поведение канваса

### 8.1. Выбор родителя

Левый клик по узлу:

- если клик пришёлся по карточке, узел выбирается как родитель по своему полу
- если узел мужской, он назначается в `fatherId`
- если узел женский, он назначается в `motherId`
- если родитель того же типа уже был выбран, он заменяется новым

Важно:

- выбранные родители автоматически подставляются в форму по созданию животного
- выбран может быть только один отец и одна мать одновременно
- валидно выбрать только одного родителя и затем нажать `Добавить`

### 8.2. Редактирование

Редактирование открывается только по клику на шестерёнку внутри карточки.

Клик по самой карточке редактирование не открывает.

### 8.3. Сброс выбранных родителей

Клик по пустому месту должен сбрасывать текущий выбор родителей.

Но:

- если пользователь тащит канвас, сброс происходить не должен
- сброс разрешён только при обычном клике по пустому полю без реального pan-жеста

### 8.4. Перемещение поля

Пустое место канваса можно перетаскивать мышью.

Требования:

- pan стартует только с пустого пространства
- есть небольшой порог начала pan, чтобы короткий клик не считался перетаскиванием
- во время pan курсор должен быть `grabbing`

Текущий порог:

- `PAN_START_THRESHOLD = 4`

### 8.5. Масштабирование

Zoom работает колесом мыши относительно позиции курсора.

Требования:

- масштаб должен быть плавным
- центр zoom должен оставаться под курсором
- масштаб ограничен диапазоном

Текущие значения:

- `MIN_ZOOM = 0.45`
- `MAX_ZOOM = 2.1`
- `ZOOM_SENSITIVITY = 0.01`

### 8.6. Кнопка сброса вида

В интерфейсе есть кнопка `Вернуться в центр` / `Back to center`.

Она должна возвращать канвас к дефолтному виду:

- `pan = { x: 180, y: 0 }`
- `zoom = 1`
- hover-состояние сбрасывается

Это поведение должно совпадать с видом после полной перезагрузки страницы.

## 9. Верхняя панель действий

В левом верхнем углу находится overlay-панель с действиями в одну линию:

- кнопка `?`
- кнопка `Вернуться в центр`
- кнопка `Добавить`

Требования:

- `?` стоит слева от остальных кнопок
- `?` открывает popup с подсказками
- `Добавить` открывает создание нового животного
- если родители выбраны заранее, форма должна открываться с уже подставленными `fatherId`/`motherId`

## 10. Popup-подсказка

Подсказки по использованию приложения не должны быть постоянно видны на канвасе.

Они находятся в отдельном popup, открываемом по `?`.

Требования к popup:

- открывается как модальное окно поверх затемнённого backdrop
- закрывается по клику на backdrop
- закрывается по крестику
- содержит короткий список правил использования

## 11. Переключатель языка

Поддерживаются только две локали:

- `ru`
- `en`

Переключатель языка расположен в правом верхнем углу.

Требования:

- хранить выбор в `localStorage`
- при загрузке пытаться взять язык из `localStorage`
- если значения нет, определять по языкам браузера
- если совпадений нет, использовать `ru`
- переключатель должен иметь собственную визуально заметную рамку и не сливаться с фоном

## 12. Легенда

В правом нижнем углу показывается legend-panel:

- male
- female

Она служит только для визуальной подсказки по цветам пола.

## 13. Форма создания/редактирования

### 13.1. Поля формы

Форма должна содержать:

- имя
- пол
- отец
- мать
- дата рождения

### 13.2. Режим создания

При создании:

- форма открывается пустой, кроме `birthDate = today`
- если родители были предварительно выбраны на канвасе, они подставляются

### 13.3. Режим редактирования

При редактировании:

- форма заполняется значениями узла
- если у узла есть потомки, поле пола блокируется
- должна быть возможность удалить узел

### 13.4. Валидация

Ошибки валидации показываются внутри формы.

Требования:

- `name` обязателен
- максимальная длина имени = `20`
- нельзя указать себя в качестве родителя
- нельзя указать одинакового отца и мать

## 14. Удаление

Удаление должно быть двухшаговым:

1. нажать кнопку удаления в форме редактирования
2. подтвердить действие в confirm-dialog

После удаления:

- узел исчезает из дерева
- у всех потомков ссылки на этого родителя очищаются

## 15. Форматирование дат

`birthDate` хранится как ISO-строка, но отображается локализованно:

- по текущему языку интерфейса
- через `Intl.DateTimeFormat`

Если дата отсутствует, должен использоваться локализованный placeholder `noDate`.

## 16. Начальные данные

Если в `localStorage` нет данных, приложение должно стартовать с демо-деревом.

Демо-данные нужны только как стартовый seed для пустого хранилища. Они не должны диктовать доменную модель приложения.

## 17. Нефункциональные требования

### 17.1. Производительность

- отрисовка дерева должна происходить в `canvas`
- layout вычисляется из массива животных
- hit-testing должен работать поверх тех же координат, что и рендер

### 17.2. Согласованность координат

Все эти операции обязаны использовать одну координатную систему:

- рендер дерева
- определение клика по узлу
- определение клика по шестерёнке
- вычисление позиции при zoom
- вычисление позиции при pan

Если эти формулы разойдутся, начнутся визуальные и UX-баги.

### 17.3. Чистота домена

- доменные сущности должны называться `Animal`, а не по конкретному виду
- нельзя возвращать видоспецифичные имена в кодовую базу
- один канвас всё ещё описывает один вид, но этот вид не кодируется в каждом узле

## 18. Структура проекта

Минимально ожидаемая структура:

- `src/App.tsx` — главный экран, канвас, модалки, взаимодействия
- `src/animal-data.ts` — типы и сериализация данных
- `src/i18n.tsx` — словари локализации и provider
- `src/styles.css` — глобальные стили

## 19. Команды разработки

```bash
npm install
npm run dev
npm run build
```

`build` должен включать:

- type-check для app-конфига
- type-check для node-конфига
- production build через Vite

## 20. Критичные подводные камни

При доработках проекта нельзя ломать следующие инварианты:

- клик по карточке и клик по шестерёнке делают разные действия
- пустой клик сбрасывает выбор, но pan не сбрасывает
- reset-view должен совпадать со стартовым видом страницы
- anchor канваса обязан использоваться везде одинаково
- импорт/экспорт в будущем должен идти через `animal-data.ts`
- пол нельзя менять, если у узла есть потомки
- `fatherId` всегда мужской, `motherId` всегда женский
- проект не хранит смешанные виды внутри одного канваса

## 21. Что считать корректной реализацией

Проект реализован правильно, если:

- дерево открывается с демо-данными и корректным layout
- канвас двигается и плавно масштабируется
- карточка и шестерёнка имеют раздельные hover-состояния
- левый клик по карточке выбирает родителя
- шестерёнка открывает редактирование
- `Добавить` создаёт новое животное с предвыбранными родителями
- `Вернуться в центр` восстанавливает дефолтный вид
- язык сохраняется между сессиями
- данные дерева сохраняются в `localStorage` в версии `1`
- сборка проходит без ошибок
